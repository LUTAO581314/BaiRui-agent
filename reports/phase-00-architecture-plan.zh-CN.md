# Phase 00 中文报告：个人与公司 Agent 系统总体架构规划

## 1. 阶段目标

本阶段目标是把 `hermes-` 从一个空仓库整理成可继续推进的规划仓库，明确整个系统未来要怎么分层、怎么部署、怎么管理公司、怎么接入微信/飞书、怎么使用 Obsidian 记忆、怎么接入搜索项目运行时和图像/视频 API，以及哪些高风险动作必须由主人确认。

## 2. 已完成内容

本阶段已经完成：

- 建立总体规划文档。
- 建立系统架构文档。
- 建立 API-first 多模态接入方案。
- 建立风险与权限边界文档。
- 建立阶段路线图。
- 优化总体架构，把系统拆成公司管理平面、个人陪伴平面、后台智能平面。
- 明确飞书优先作为公司管理入口。
- 明确微信后续作为个人陪伴和轻量提醒入口。
- 明确轻量 VPS 只做调度、消息适配、日志、缓存、API 调用和 Obsidian 写回，不跑本地大模型。
- 明确以后每个阶段都要给主人写中文报告。

## 3. 当前效果

当前仓库已经不是空壳，而是一个可以继续交给 Agent 或工程师执行的架构蓝图。

它现在能回答这些问题：

- Hermes 负责什么。
- Obsidian 负责什么。
- 飞书负责什么。
- 微信负责什么。
- BaiLongma 放在什么位置。
- MiroFish 放在什么位置。
- 搜索、图像识别、视频理解怎么接入。
- 轻量服务器能做什么，不能做什么。
- 公司管理第一版应该先做什么。
- 哪些操作必须经过主人确认。

当前还没有完成真实运行部署，所以它还不能实际在飞书里管理公司，也不能自动写 Obsidian 报告。下一阶段需要进入部署和最小可用系统建设。

## 4. 优化后的总体架构

优化后的架构分为三层：

```text
公司管理平面
  飞书
  项目、客户、销售、回款、日报、审批、会议、风险

个人陪伴平面
  微信 / BaiLongma
  主动聊天、个人提醒、快速记录、轻量摘要

后台智能平面
  Hermes
  搜索、研究、API 调用、任务调度、日志、Obsidian 写回
```

### 4.1 飞书的定位

飞书是第一生产入口，适合管理公司。

它应该负责：

- 项目管理。
- 客户跟进。
- 销售漏斗。
- 回款提醒。
- 员工日报。
- 会议纪要。
- 审批提醒。
- 风险台账。
- 每日经营简报。

飞书比微信更适合公司管理，因为它有更清晰的组织、权限、任务、文档、多维表格和审批体系。

### 4.2 微信的定位

微信不建议作为第一版公司管理入口。

微信更适合：

- 主动陪主人聊天。
- 做个人提醒。
- 接收临时想法。
- 推送少量重要提醒。
- 作为后续 BaiLongma 的个人互动入口。

涉及公司人员、审批、回款、合同、对外承诺的动作，不建议放在微信里直接执行。

### 4.3 Hermes 的定位

Hermes 是后台总管，不是单纯聊天机器人。

它应该负责：

- 定时任务。
- 搜索研究。
- 飞书数据分析。
- 多模态 API 调用。
- 公司风险检查。
- 经营日报生成。
- Obsidian 写回。
- 审批前的草稿和建议。

### 4.4 Obsidian 的定位

Obsidian 是长期记忆库。

它应该保存：

- 主人偏好。
- 公司经营报告。
- 决策记录。
- 项目复盘。
- 重要客户背景。
- 研究报告。
- MiroFish 推演结果。
- Agent 行为总结。

飞书是工作的发生地，Obsidian 是长期记忆和复盘地。

### 4.5 MiroFish 的定位

MiroFish 不做主记忆库，而做推演沙盘。

它适合：

- 项目决策模拟。
- 股票/市场多空推演。
- 商业策略比较。
- 风险机会分析。
- 多 Agent 辩论和报告。

## 5. API-first 决策

服务器是轻量 VPS，不适合跑大模型、图像模型或视频模型。

因此第一版采用 API-first：

- 搜索优先走 TrendRadar、SearXNG 这类外部项目运行时。
- 图片理解走视觉模型 API。
- OCR 走 OCR API 或多模态 API。
- 语音转写走 Whisper-compatible API。
- 视频理解走视频 API，或“语音转文字 + 抽帧看图”的组合流程。

服务器只负责：

- Hermes 调度。
- 飞书/微信适配。
- API 调用。
- 队列和重试。
- 缓存。
- 日志。
- Obsidian 写回。

这个方案更适合当前服务器资源，部署快，风险低，也方便以后更换模型供应商。

## 6. 公司管理第一版建议

第一版不要一口气接所有功能，建议先做飞书公司管理 MVP。

建议最小版本：

```text
Hermes on VPS
+ 飞书公司管理机器人
+ 飞书多维表格
+ Obsidian 经营报告
+ 搜索/API 调用能力
+ 主人审批队列
```

第一批飞书表格建议：

- 项目表。
- 客户表。
- 销售漏斗表。
- 回款表。
- 员工日报表。
- 风险台账。

第一批自动化建议：

- 每日早晨经营简报。
- 中午异常提醒。
- 晚上公司总结。
- 逾期任务提醒。
- 客户跟进提醒。
- 回款风险提醒。
- 主人审批队列。

## 7. 风险与边界

必须保持以下边界：

- 不能让 Agent 未经确认自动下单交易。
- 不能让 Agent 未经确认审批大额费用。
- 不能让 Agent 未经确认开除员工、改薪资、签合同。
- 不能让 Agent 未经确认对外发送承诺。
- 不能把 API key、密码、私钥提交到 GitHub。
- 微信个人号桥接要谨慎，不能作为第一阶段公司管理主通道。

默认允许：

- 查资料。
- 总结。
- 写报告。
- 提醒。
- 草拟文档。
- 检查延期和风险。
- 给出建议。

默认需要主人确认：

- 花钱。
- 交易。
- 对外承诺。
- HR 处理。
- 合同变更。
- 服务器破坏性操作。

## 8. 需要主人确认的事项

下一阶段开始前，需要主人后续确认：

- 模型 API 用哪一家。
- 是否有域名，没有也可以先用 IP + 安全回调方案。
- 飞书公司主体和应用是否已经准备好。
- 第一版是否只做飞书，不做微信。
- Obsidian vault 放在 VPS 上还是主人本地同步。
- 公司管理第一批表格字段是否按本报告建议来。
- 是否要先做经营日报，还是先做任务/项目追踪。

## 9. 下一阶段计划

建议下一阶段是 Phase 01：Hermes VPS MVP + 飞书公司管理基础。

目标：

1. 在 VPS 上部署 Hermes 或最小调度服务。
2. 建立非 root 运行用户。
3. 建立 `.env` 配置模板。
4. 接入一个模型 API。
5. 准备飞书机器人回调。
6. 建立飞书公司管理表格。
7. 生成第一份飞书经营简报。
8. 把经营简报写入 Obsidian。
9. 给主人提交 Phase 01 中文报告。

## 10. 当前结论

当前架构已经从“个人 Agent 工具集合”升级成“个人 + 公司 Agent 操作系统”。

最稳的推进路线是：

```text
先飞书公司管理
再 Obsidian 长期记忆
再 API 多模态能力
再微信个人陪伴
再 BaiLongma 人格层
再 MiroFish 推演室
最后才考虑交易或高风险执行
```

这样做可以最快产生实际管理价值，同时避免一开始就把微信、交易、视频模型、本地大模型和公司权限全部混在一起，导致系统复杂且难以控制。
