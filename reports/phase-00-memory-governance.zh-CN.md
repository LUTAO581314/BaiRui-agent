# Phase 00 补充中文报告：类人联想记忆与 Obsidian 可视化方案

## 1. 主人的担心

主人担心的是对的：如果 Agent 什么都记，最后会变成一堆杂乱、重复、过期、错误、无价值的垃圾记忆。

更重要的是，**不能只按时间记忆**。

按时间记忆只是流水账。人类真正有用的记忆不是按“几点几分说了什么”堆起来，而是通过关系、重要性、重复、情绪、目标、人物、项目、场景和因果关联形成网络。

所以这个系统的记忆不能设计成简单日志库，而应该设计成：

```text
类人联想记忆
+ Obsidian 双链图谱
+ 定期清理
+ 主人可纠错
+ 机器索引辅助
```

## 2. 核心结论

记忆系统应该遵循一句话：

```text
少记、精记、有关联、可视化、可纠错、可过期。
```

时间只作为元信息，不作为唯一主轴。

真正的主轴应该是：

- 人。
- 项目。
- 公司。
- 客户。
- 目标。
- 决策。
- 风险。
- 事件。
- 反复出现的问题。
- 主人的偏好。
- 重要报告和复盘。

## 3. 人是怎么记忆的

人不是把所有东西按时间完整存档。

人更像这样记：

```text
这个人是谁
这件事和哪个项目有关
它为什么重要
它让我以后要避免什么
它和之前哪次经历相似
它导致了什么结果
它是否反复出现
它会不会影响当前目标
```

所以 Agent 也应该这样记：

```text
事实 + 关系 + 来源 + 重要性 + 复盘价值
```

而不是：

```text
时间 + 原始聊天记录
```

## 4. Obsidian 为什么适合

Obsidian 适合做主记忆库，不只是因为它能写 Markdown，而是因为它有：

- 双链。
- 反向链接。
- 标签。
- 属性/frontmatter。
- 全局图谱。
- 局部图谱。
- Canvas 手动画布。
- MOC 主题索引笔记。

这些能力能把“记忆”变成可视化网络。

例如：

```text
[[客户A]] -> [[项目X]] -> [[回款风险]] -> [[2026-W23周报]]
[[主人偏好]] -> [[中文阶段报告]] -> [[报告规范]]
[[股票观察]] -> [[多头理由]] -> [[风险触发条件]]
```

这比“2026-06-06 说过某句话”有用得多。

## 6. 记忆不是日志

日志可以多，记忆必须少。

日志记录发生了什么，记忆记录“以后有用的东西”。

例如：

```text
日志：今天 10:32 搜了某个网页
记忆：AlphaX 风险较高，只适合作为观察对象，不适合作为核心系统
```

```text
日志：某员工今天日报迟交
运营记忆：该员工本周两次迟交日报
长期记忆：如果连续多周出现，再记录为管理风险
```

## 7. 每条长期记忆都要有关系

写入 Obsidian 的长期记忆，至少应该链接到一个关系轴：

- 相关人物。
- 相关项目。
- 相关目标。
- 相关客户。
- 相关决策。
- 相关报告。
- 相关风险。

如果一条记忆没有任何关系，它大概率只是垃圾碎片。

可以先放到：

```text
00-Inbox/needs-review
```

等待整理，不要直接污染长期记忆。

## 8. Obsidian 可视化怎么玩

### 8.1 全局图谱

用来看整个系统的结构：

- 哪些项目是核心。
- 哪些客户和项目有关。
- 哪些报告反复引用同一个风险。
- 哪些记忆孤立无连接。

孤立节点通常要处理：

- 有价值就补链接。
- 没价值就归档或删除。

### 8.2 局部图谱

用来看一个具体对象的上下文：

- 一个客户周围有哪些项目、报价、风险、会议。
- 一个项目周围有哪些人、任务、决策、报告。
- 一个主人偏好影响了哪些流程。

### 8.3 Canvas

Canvas 适合做手工整理的“大图”：

- 公司运营地图。
- 个人记忆地图。
- Agent 系统架构图。
- 投研决策流程图。
- 客户关系图。

Canvas 是人工 curated 的，适合给主人看。

### 8.4 MOC 主题索引

MOC 就是主题入口笔记。

例如：

- `company-memory-map.md`
- `project-memory-map.md`
- `market-memory-map.md`
- `decision-memory-map.md`

它们不是流水账，而是给主人快速进入某个记忆领域的导航。

## 9. 什么不能乱记

默认不应该进入长期记忆：

- 闲聊噪音。
- 重复问候。
- 临时草稿。
- 未验证猜测。
- 过期任务状态。
- 没有结论的网页摘录。
- 重复 API 输出。
- 主人一时情绪化但没有明确要求记住的话。
- 密码、API key、私钥。
- 身份证、银行卡、财务账号截图等敏感信息。

这些最多进入临时上下文、日志、日报或待审核 inbox。

## 10. 记忆写入前的八个问题

Agent 写长期记忆前，必须问：

1. 任务结束后还重要吗？
2. 这件事稳定吗？
3. 来源清楚吗？
4. 是否包含隐私或敏感信息？
5. 是否和已有记忆重复？
6. 它应该是事实，还是只应该放进报告？
7. 它应该连接到哪个人、项目、目标、决策或事件？
8. 是否需要主人批准再保存？

如果回答不上来，就不要写长期记忆。

## 11. 清理机制

建议设置三种清理节奏。

### 每周清理

- 合并重复 inbox。
- 把零散日志总结成周报。
- 删除失败或无价值 API 输出。
- 归档已完成任务草稿。
- 找孤立节点，决定补链接还是归档。

### 每月清理

- 检查过期项目状态。
- 检查客户跟进状态。
- 检查主人偏好是否有变化。
- 归档不活跃项目。
- 标记失效链接和旧研究。
- 检查图谱里是否出现垃圾中心节点。

### 每季度清理

- 重建搜索索引。
- 检查敏感记忆。
- 确认长期指令是否仍然有效。
- 检查高频被引用的记忆是否准确。

## 12. 主人如何纠错和删除

主人以后可以直接说：

```text
这条记忆错了
忘掉这个
以后不要记这种
把这个合并到那个项目
这个只放日报，不要当长期事实
把这个和那个客户连起来
```

系统应该做到：

1. 找到相关记忆。
2. 修改、合并、归档、删除或补链接。
3. 必要时记录一条纠错记录。
4. 更新搜索索引。
5. 告诉主人改了什么。

主人必须永远有最终编辑权。

## 13. 机器索引的位置

向量库、Meilisearch、SQLite FTS、Zep 都只是辅助索引。

它们不能成为最终真相。

真正长期可信的记忆应该是：

```text
Obsidian 可读文件
飞书结构化表格
主人明确确认
```

索引可以删掉重建，记忆本体不能藏在黑盒里。

## 14. 当前建议

下一阶段部署前，应该把记忆治理当成基础设施一起做。

第一版就应该有：

- Obsidian 双链结构。
- inbox 缓冲区。
- MOC 主题地图。
- Canvas 可视化地图。
- 主人纠错入口。
- 定期清理任务。
- 搜索索引重建机制。
- 敏感信息不入库规则。

## 15. 结论

主人不用担心“记忆多了必然变垃圾”。

真正的问题不是记忆多，而是没有结构、没有筛选、没有可视化、没有清理。

最好的方案不是时间线记忆，而是：

```text
时间做索引
关系做主干
Obsidian 做图谱
报告做沉淀
索引做检索
主人做最终裁判
```

这样 Agent 会越来越懂主人，而不是越来越乱。

## 16. 后续总控规则

本报告是 Phase 00 的记忆治理补充报告。后续所有阶段的记忆写入、整理、归档、删除、索引重建和中文报告，统一纳入 `docs/SUSTAINABLE_ITERATION_BLUEPRINT.md` 定义的可持续迭代循环。

每个阶段结束时都要检查：

- 哪些内容进入长期记忆。
- 哪些内容只保留在阶段报告。
- 哪些日志应该归档或删除。
- 哪些 Obsidian 链接、MOC 或 Canvas 需要更新。
- 是否需要重建搜索或向量索引。
