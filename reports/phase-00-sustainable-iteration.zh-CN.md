# Phase 00 补充中文报告：可持续迭代完整方案

## 1. 阶段目标

本报告的目标是把当前仓库里的架构、记忆、飞书、微信、API、安全、中文报告规则统一成一套可持续迭代方案。

主人要的不是一次性搭一个机器人，而是一个能长期进化、可验证、可复盘、可清理、可扩展的个人与公司 Agent 操作系统。

## 2. 总体方案

系统分为三个稳定平面：

```text
公司管理平面：飞书
个人陪伴平面：微信 / BaiLongma
后台智能平面：Hermes
```

配套基础设施：

```text
Obsidian = 长期记忆与复盘
API = 搜索、图像、视频、语音、行情等能力
MiroFish = 推演和决策沙盘
GitHub = 架构、部署、规则和报告的版本源
```

## 3. 可持续迭代循环

以后每个阶段都按同一个循环执行：

```text
阶段简报
-> 锁定范围
-> 实施或文档更新
-> 验证
-> 中文报告
-> 记忆整理
-> 风险复核
-> 下一阶段决策
```

这样可以避免系统越做越散，也避免一口气堆太多组件导致不可控。

## 4. 每阶段必须满足的门槛

每个阶段完成前必须过六道门槛：

1. 功能门槛：承诺的东西真的能跑，或者明确说明还没跑通。
2. 安全门槛：没有密钥泄露，高风险动作必须主人确认。
3. 记忆门槛：有价值输出写入 Obsidian，垃圾日志不进长期记忆。
4. 文档门槛：架构或计划变化后，同步更新对应文档。
5. 中文报告门槛：给主人一份中文阶段报告。
6. GitHub 门槛：仓库改动提交并推送。

## 5. 推荐阶段顺序

### Phase 1：VPS 运行基础

目标：

- 让服务器具备安全运行 Hermes 或最小调度服务的能力。

交付：

- 非 root 用户。
- Docker 或服务运行器。
- 环境变量模板。
- 健康检查。
- 日志。
- 启停命令。
- Phase 01 中文报告。

### Phase 2：飞书公司管理 MVP

目标：

- 让飞书成为第一个真正有用的生产入口。

交付：

- 飞书应用。
- 公司管理机器人。
- 项目表。
- 客户表。
- 销售漏斗表。
- 回款表。
- 日报表。
- 风险表。
- 早晨经营简报。
- 主人审批队列。
- Phase 02 中文报告。

### Phase 3：Obsidian 记忆 MVP

目标：

- 建立可视化、可纠错、可清理的长期记忆。

交付：

- vault 结构。
- inbox 缓冲区。
- MOC 主题地图。
- Canvas 记忆地图。
- 记忆写入规则。
- 纠错和删除流程。
- 每周清理流程。
- Phase 03 中文报告。

### Phase 4：API 多模态能力

目标：

- 通过 API 接入搜索、抓取、OCR、图像、语音、视频能力。

交付：

- 搜索适配器。
- 网页抓取适配器。
- OCR/图像理解适配器。
- 语音转写流程。
- 视频总结流程。
- 成本和限流控制。
- Obsidian 写回模板。
- Phase 04 中文报告。

### Phase 5：飞书工作流强化

目标：

- 从“能发消息的机器人”升级成“可靠的公司运营助手”。

交付：

- 任务提醒。
- 日报/周报。
- 会议纪要。
- 异常提醒。
- 审批边界。
- 审计日志。
- Phase 05 中文报告。

### Phase 6：投研和机会雷达

目标：

- 建立只读、研究型的机会发现系统。

交付：

- 观察池。
- 行情数据。
- 新闻/研究来源。
- 多空分析。
- 风险清单。
- 飞书摘要。
- 不接真实交易。
- Phase 06 中文报告。

### Phase 7：BaiLongma/个人互动层

目标：

- 加入中文人格、主动关怀、个人提醒和快速记录。

交付：

- 中文交互入口。
- 个人记忆边界。
- 快速记录。
- 主动提醒。
- Phase 07 中文报告。

### Phase 8：微信桥接评估

目标：

- 只在风险可接受的情况下接入微信。

交付：

- 微信风险报告。
- 官方通道优先方案。
- 限流和确认机制。
- 敏感命令禁用。
- Phase 08 中文报告。

### Phase 9：MiroFish 推演室

目标：

- 做项目、市场、商业策略的多 Agent 推演。

交付：

- 推演输入模板。
- Obsidian 导出。
- MiroFish 推演。
- 报告导回 Obsidian。
- Phase 09 中文报告。

### Phase 10：运维强化

目标：

- 让系统可维护、可恢复。

交付：

- 备份。
- 恢复手册。
- 日志轮转。
- 健康监控。
- 告警。
- 密钥轮换。
- 索引重建。
- Phase 10 中文报告。

### Phase 11：高风险执行集成

目标：

- 只有在安全设计完成后，才考虑真实世界执行 API。

可能包括：

- 券商 API。
- 支付。
- 云服务写操作。
- 公开发布。

默认全部禁用，除非有单独威胁模型、审批流、限额、审计日志、kill switch 和中文风险报告。

## 6. 记忆如何保持干净

每个阶段结束都要做记忆整理：

- 哪些内容变成长期记忆。
- 哪些只保留在阶段报告里。
- 哪些日志归档。
- 哪些临时内容删除。
- 哪些 Obsidian 链接需要补。
- 哪些 Canvas/MOC 需要更新。
- 搜索或向量索引是否要重建。

记忆不是按时间线乱堆，而是按关系组织：

```text
人、项目、目标、客户、决策、风险、报告、事件、重复模式
```

## 7. 风险边界

默认允许：

- 查资料。
- 总结。
- 写报告。
- 发给主人提醒。
- 草拟文档。
- 检查延期和风险。
- 生成建议。

默认禁止：

- 自动交易。
- 自动花钱。
- 自动签合同。
- 自动对外承诺。
- 自动处理 HR 敏感事项。
- 自动删除重要数据。
- 把密钥写进 GitHub。
- 微信个人号高风险自动化。

## 10. 结论

现在系统已经有了可持续迭代方法。

不是“今天想到什么就加什么”，而是：

```text
小步交付
每步验证
每步中文报告
每步整理记忆
每步复核风险
每步更新仓库
```

这样它才能长期成长，而不是堆成一个难维护的工具集合。
